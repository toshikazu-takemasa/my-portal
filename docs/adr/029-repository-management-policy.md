# ADR-029: リポジトリ管理方針（AI基盤進化とADR駆動の統合）

## ステータス
承認済み (Accepted)

## コンテキスト
本プロジェクトは、個人のポータルアプリとしての利便性を高めつつ、将来的に「AIフレンドリーな知識基盤（ADR-028）」へと進化することを目指している。リポジトリの成長に伴い、機能が場当たり的に実装され、データの構造や設計意図がブラックボックス化（サイロ化）することを防ぐ必要がある。

## 決定事項
「次世代AI基盤（プラン3）」への進化を方向性とし、「ADR駆動（プラン2）」を管理手法として採用する。具体的には以下の原則を遵守する。

### 1. ADRによる設計・データ構造の先行定義
- 全ての構造的変更、データスキーマの変更、および重要な機能追加は、実装前にADRに記録する。
- ADRでは、UIの実装詳細よりも「どのようなデータ形式（Markdown Frontmatter, JSON Schema）で情報を管理するか」を優先的に定義する。

### 2. データ中心のサイロ化防止策
- **相互運用性の確保**: 特定のUI実装に依存しないデータ形式を採用し、将来のNext.js移行やMCPサーバーからの参照を容易にする。
- **メタデータの標準化**: AIが文脈を即座に理解できるよう、データの命名規則とメタデータ（Frontmatter）の付与を徹底する。

### 3. 進化的アーキテクチャの採用
- **移行経路の確保**: 現在の Vanilla JS での実装においても、将来の API Routes への移植やコンポーネント化を見据えたロジックの分離（関心の分離）を意識する。
- **機能フラグによる制御**: 仕事・個人の切り替えや、実験的機能の導入はフラグ管理を検討し、一つのコードベースが肥大化・複雑化するのを防ぐ。

### 4. 文脈の連結（Context Linking）
- 各ADRは関連する過去のADRを参照し、設計の進化過程を明確にする。
- データの依存関係を明示し、ある変更が他の領域に与える影響をAIと人間が把握しやすくする。

## 意図される結果
- リポジトリ全体が「AIが理解可能な知識ベース」として一貫性を保つ。
- データのポータビリティが確保され、特定の技術スタック（現在のVanilla JS等）に縛られない資産となる。
- 実装の「理由」が常にドキュメント化されているため、開発のブランクがあっても再開が容易になる。
