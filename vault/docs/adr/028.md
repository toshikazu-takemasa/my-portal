AI-Friendly Portal: GitHub-as-a-Backend 設計仕様書

1. 背景と目的

従来のリポジトリ（my-portal）は、GitHubを知識ベースのファイルシステムとして利用することで、AI（LLM）による構造的な理解を容易にすることを目的としていた。しかし、GitHub PAT（個人用アクセストークン）の生成やGitの概念理解が必要であり、一般ユーザーにとっての導入障壁が高いという課題があった。

本ドキュメントでは、**「AIフレンドリーな構造化データ」と「ユーザーフレンドリーなUX」**を両立するための、GitHub Appを活用した次世代ポータルアプリの設計案を提示する。

2. コア・アーキテクチャ

ユーザーにGitを意識させず、裏側でGitHubを堅牢なストレージとして運用する「Local-first + Auto-sync」モデルを採用する。

A. 認証レイヤー (Auth Layer)

GitHub App (OAuth): PATの代わりにGitHub Appによる認可を使用。

Repository Selection: ユーザーは「全リポジトリ」ではなく、「アプリ専用の特定リポジトリ」のみへのアクセスを許可できるため、セキュリティの不安を解消する。

B. データ同期レイヤー (Sync Layer)

Octokit.js: GitHub API（Contents API）を使用して、ブラウザから直接、またはAPI Route経由でファイルを読み書きする。

Auto-Provisioning: 初回ログイン時、アプリ専用のプライベートリポジトリを自動作成する。

C. UXレイヤー (UX Layer)

No-Git UI: 「コミット」「プッシュ」という用語を排除し、「保存中」「同期完了」という表現に統一。

Local Cache: IndexedDB等に一時保存し、オフライン環境でも書き込みを可能にする。

4. 実装フロー (PoC)

1. ユーザー認可と初期設定

ユーザーが「GitHubでログイン」を実行。

アプリがユーザーのアカウント内に my-ai-knowledge-base（仮）というプライベートリポジトリの有無を確認。

存在しない場合、GitHub API経由でリポジトリを自動生成。

初期設定ファイル（README.mdやAI向けの instruction.md）を自動配置。

2. データ保存の自動化

ユーザーが日記やタスクを入力・保存する際：

ブラウザ側: 内容をMarkdown形式に変換。

メタデータ付与: YAML Frontmatter（作成日時、感情タグ、コンテキスト等）を自動挿入。

同期: Octokitを用いて journal/2026/05/04.md のようなパスで自動コミット。

5. AIフレンドリーであるための工夫

単にファイルを置くだけでなく、AIが「読みやすく、活用しやすい」ためのデータ構造をアプリ側で保証する。

セマンティックなディレクトリ構造:

/journal/YYYY/MM/DD.md : 日記データ

/tasks/active.json : タスクデータ

/config/ai-persona.md : AIへの指示書

Frontmatterの標準化:
各Markdownの冒頭に、AIが瞬時に文脈を把握できるメタデータを付与する。

---
date: 2026-05-04
mood: positive
tags: [ideation, architecture]
---



MCP (Model Context Protocol) への対応:
将来的にリポジトリをAIツール（Claude Desktop等）に直接接続できるMCPサーバーとして機能させるためのメタデータを含める。

MCP (Model Context Protocol) 連携によるAIポータルの深化

1. MCP導入のメリット：AIを「閲覧者」から「編集者」へ

現在の「GitHubにファイルを置く」だけでは、ユーザーがAIにファイルをアップロードするか、AIがGitHubリポジトリ全体をスキャンする必要があります。
MCPを導入すると、AI（例：Claude Desktop）はあなたの指示に応じて、必要な情報だけをピンポイントで取得したり、タスクを完了させたりできるようになります。

実現できる体験

AIへの問いかけ: 「先週のジャーナルから、体調に関する記述を抽出して要約して」

AIによる操作: 「今日のタスクに『プロテインを買う』を追加しておいて」

AIによる分析: 「最近のタスク完了率をグラフ化するためのデータを整理して」

2. MCPサーバーの3つの構成要素

「My Portal」専用のMCPサーバーを構築する場合、以下の3つの機能をAIに提供します。

① Resources（リソース）: データの閲覧

AIが読み取り専用でアクセスできるデータ。

Journal Resource: 日付を指定して日記の内容をテキストとして提供。

Task List Resource: 現在の active.json の内容を構造化データとして提供。

② Tools（ツール）: データの操作

AIが「実行」できる関数。

add_task(title, priority): GitHub上のJSONを更新する関数。

search_journal(query): 日記の内容をキーワード検索する関数。

finalize_day(): その日の日記とタスクを元に、日報を自動生成してGitHubに保存する関数。

③ Prompts（プロンプト）: 定型アクション

AIに対する「定型的な依頼」のテンプレート。

"Weekly Review": 「先週の全ジャーナルを読み、次の1週間の目標を提案してください」という指示セットをAIに即座に与える。

3. 実装アプローチ

アプローチA：既存のGitHub MCPサーバーを利用（即時利用）

Anthropicが公開している公式の github-mcp-server を利用します。

メリット: 開発不要。

デメリット: AIは「GitHubのファイル操作」として認識するため、「日記」や「タスク」というアプリ固有の意味を理解させるには、毎回長いプロンプトが必要です。

アプローチB：専用MCPサーバーを構築（推奨）

Node.js (TypeScript) + MCP SDK を使用して、あなたのアプリ専用の「ブリッジ」を作ります。

AI-Friendly Portal: GitHub-as-a-Backend 設計仕様書

1. 背景と目的

従来のリポジトリ（my-portal）は、GitHubを知識ベースのファイルシステムとして利用することで、AI（LLM）による構造的な理解を容易にすることを目的としていた。しかし、GitHub PAT（個人用アクセストークン）の生成やGitの概念理解が必要であり、一般ユーザーにとっての導入障壁が高い。

本ドキュメントでは、「AIから容易に参照可能な知識ベース」を、GitHub Appを活用してユーザーフレンドリーに構築するための設計案を提示する。

2. コア・アーキテクチャ

ユーザーには使いやすいエディタ（UI）を提供し、裏側でGitHubを「AIフレンドリーな構造化ストレージ」として運用する。

A. 認証・認可 (Auth Layer)

GitHub App (OAuth): ユーザーにPATを生成させず、「GitHubでログイン」のみで連携を完了させる。

Scoped Access: アプリ専用のリポジトリのみに権限を絞り、セキュリティと安心感を提供する。

B. データの「通訳」レイヤー (Access Bridge)

GitHub APIの抽象化: GitHubの生データを直接AIに渡すのではなく、Vercel/Cloudflare上で「AIが解釈しやすい形式」に変換して提供する窓口（API/MCP）を配置する。

C. データの「標準化」 (Storage Layer)

Markdown + JSON: AIが最も高い精度で扱える形式で保存。

Auto-Provisioning: 初回ログイン時にAIが読み取りやすいディレクトリ構造（journal, tasks, config）を自動構築する。

3. 推奨インフラ・技術スタック

4. AIからの容易な参照を実現する「Model-Agnostic」設計

特定のAIサービスに依存せず、将来登場するあらゆるAIがあなたのデータを利用できる仕組みを構築する。

A. MCP (Model Context Protocol) サーバーの提供

データへの窓口をMCP規格で公開することで、Claude DesktopやCursor、IDEなどの「MCP対応AIクライアント」が、あなたのGitHubデータを直接「自分の記憶」として参照可能になる。

B. OpenAPI 仕様の提供

MCP非対応のAI（GPTsのActionsなど）でも利用できるよう、標準的なREST APIとしてデータ取得口を構築する。

C. プレーンテキストの徹底

独自のバイナリ形式や複雑なDBを使わず、MarkdownとJSONという「AIにとっての共通言語」を徹底する。

5. 実装フロー (PoC)

認可: ユーザーがログインし、アプリ専用リポジトリへのアクセスを許可。

自動構築: アプリがリポジトリ内に /journal, /tasks, /meta 等のディレクトリを自動作成。

保存: ユーザーが日記を書くと、メタデータ（Frontmatter）付きMarkdownとして自動保存。

露出: Vercel等で動くAPI/MCP経由で、外部のAIがこれらのファイルを構造的に読み取る。

6. 結論

本設計は「AIを作る」ことではなく、**「あなたの人生のデータをAIが自由に、かつ安全に活用できる状態にする」**ことを目的とする。GitHubという堅牢なストレージと、Vercel/Cloudflareという軽量なブリッジを組み合わせることで、ユーザーに負担をかけない「次世代の知識管理システム」を実現する。
